# Pedagogical Documentation

## What I do

- Guide the writing of clear, progressive and student-friendly technical documentation.
- Help structure explanations, examples, exercises and solutions so learners can follow the workshop step by step.
- Keep the tone, terminology and teaching flow consistent across the material.

## When to use me

- Use me when creating or revising guides, theory, exercises or reference solutions for students.
- Use me when documentation needs to be clearer, more progressive or better aligned with the workshop's teaching goals.

## Main Goal

Write documentation that helps programming students understand the workshop step by step without assuming prior mastery of MongoDB, KMongo or Kotlin.

## Writing Principles

- Start from the learning objective before showing code.
- Explain one concept at a time.
- Prefer short paragraphs and direct sentences.
- Address the student directly in second person whenever it improves clarity.
- Use consistent terminology across the whole workshop.
- Avoid unnecessary jargon; if a technical term is needed, define it briefly.
- Make the progression explicit: what the student already knows, what they will do next and why it matters.

## Teaching Flow

When documenting a new section, follow this order whenever possible:

1. What the section teaches.
2. Why it is useful in a MongoDB application.
3. Minimal Kotlin/KMongo example.
4. Explanation of the example.
5. Common mistakes or warnings.
6. Small exercise or practice prompt.

## Exercises and Solutions

- State the objective clearly.
- Indicate the expected input, output or result.
- Keep exercises aligned with the code already explained.
- Mark solutions clearly with a visible heading.
- In solutions, explain the key decision, not every trivial line.

## Style Rules
- Use spanish of spain, not latin american spanish, for all documentation, exercises and solutions, to maintain consistency with the rest of the documentation and the target audience.
- Use a professional but approachable tone.
- Prefer direct phrasing aimed at the student, such as `vas a aprender`, `usarás`, `puedes` or `te conviene`.
- Avoid impersonal formulas such as `el alumnado`, `se usará`, `se trabajará` or `la idea didáctica es` when a direct second-person sentence works better.
- Prefer active voice.
- Avoid overly long lists.
- Use examples grounded in the workshop domain, such as books, users and loans.
- Do not include hidden assumptions that force the learner to infer missing setup.
- Professional formatting: use code blocks for code, bullet points for lists, and headings to structure content.
- Use consistent formatting for code snippets, exercises and solutions.
- Use comments in code snippets to explain the purpose of key lines or sections.
- When referring to files, classes or variables, ensure they are defined in the documentation or codebase.
- When introducing a new concept, provide a brief definition or link to a resource for further reading.
- When explaining a code snippet, break it down into smaller parts and explain each part separately before showing the whole snippet.
- When providing exercises, ensure they are directly related to the content just explained and do not require knowledge of concepts that have not yet been covered.
- When providing solutions, focus on the key steps and decisions rather than explaining every line of code, to encourage learners to understand the logic rather than just copy the solution.

## Things to Avoid

1. Explaining advanced abstractions too early.
2. Mixing exercise statements with solutions.
3. Referring to files, classes or variables that do not exist.
4. Using ambiguous instructions like "do the usual setup".
5. Copying Python wording when the Kotlin equivalent needs a different explanation.
6. Assuming the learner knows how to run a Kotlin application without providing instructions.
7. Using inconsistent terminology (e.g., switching between "collection" and "table" without explanation).
8. Providing solutions that are too verbose and explain every line, which can overwhelm learners.
9. Using technical jargon without explanation.
10. Providing exercises that require knowledge of concepts not yet covered in the documentation.
11. Using a tone that is too formal or too casual, which can alienate learners.
12. Providing code snippets without comments or explanations, which can make it difficult for learners to understand the purpose of the code.
13. Using inconsistent formatting for code snippets, exercises and solutions, which can make the documentation harder to follow.
14. Referring to files, classes or variables that are not defined in the documentation or codebase, which can lead to confusion.
15. Introducing new concepts without providing a brief definition or link to further resources, which can leave learners feeling lost.
